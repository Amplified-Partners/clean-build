# Dev Throughput & AI Usage Guide

## 1. Purpose

This document defines realistic daily/weekly development capacity for the AI-first solo-founder studio, and how to use the AI stack (Cursor + Kilo Code + Claude + ChatGPT + local LLMs) to hit that capacity without trashing code quality.

This is about **merged, tested, production-quality code**, not raw code generation.

---

## 2. Key Definitions

**Net LoC (lines of code):**  
`LoC added – LoC removed` in merged PRs on `main` (or the primary trunk branch).

**Feature slice (“profit slice”):**  
A small, testable unit of value that can be spec’d, implemented, tested, and deployed in 0.5–2 days. It should move a metric or contribute directly to a revenue/validation path.

Typical feature slice impact (after cleanup):

- Frontend: ~80–250 LoC (components, hooks, types, tests).
- Backend: ~80–250 LoC (routes, handlers, types, tests).
- Config/docs: ~20–80 LoC.

So a typical slice is **~200–500 net LoC** once merged.

---

## 3. Realistic Daily Capacity

Assumptions:

- You use **Cursor + Kilo Code** as the main coding environment.
- Coding models: **Claude Code** + **ChatGPT code models** (+ local LLMs when useful).
- Every change is:
  - Tied to a spec / feature note.
  - Implemented on a branch.
  - Passing tests/linters.
  - Merged via PR with CI.

Under these conditions, a **realistic, healthy daily capacity** is:

> **200–800 net LoC/day** (merged, tested, reviewed)  
> **per focused founder**

Some days will be **0 or negative net LoC** when doing refactors/cleanup — that’s expected and healthy.

---

## 4. Operating Modes

### 4.1 Mode A – Exploration / Prototyping

Goal: learn fast, throw away aggressively.

- Output:
  - 1,000–3,000+ LoC/day is easy with AI.
  - Most of it should be considered disposable.
- Typical activities:
  - Trying new libraries/stacks.
  - Spiking an idea for feasibility.
  - Testing UX/interaction patterns.

**Rule:** Prototype in isolated branches or sandboxes. Only promote a *small, cleaned-up subset* into main.

---

### 4.2 Mode B – Profit Slices (Primary mode)

Goal: ship small, profitable or validation slices into production.

- Output target:
  - **1–2 feature slices/day.**
  - **200–800 net LoC/day** merged.
- Activities:
  - Refining the feature/spec note.
  - Implementing frontend/backend code.
  - Writing or updating tests.
  - Updating docs and Obsidian notes.
  - PR review and CI fixups.

**This is the mode to optimise for.**

---

### 4.3 Mode C – Refactor / Infra / Cleanup

Goal: keep the system fast, clean, and maintainable.

- Output:
  - Net LoC often **0 or negative**.
  - You might touch 500–1,500 lines; many will be removed.
- Activities:
  - Extract shared modules.
  - Delete dead code.
  - Improve architecture boundaries.
  - Upgrade dependencies.
  - Improve logging/observability.

**Rule:** A day with fewer lines and a better architecture is a success.

---

## 5. How AI Multiplies Throughput (Without Killing Quality)

AI does not matter because it adds more lines. It matters because it lets you do **more loops** per unit time.

Without AI (traditional solo dev):

- ~1 feature slice every 1–3 days.
- ~100–300 LoC/feature.
- Very limited experimentation.

With AI (Cursor + Kilo Code + Claude Code + ChatGPT code):

- Same complexity feature in **~½–⅓ the time**.
- 1–2 slices/day is realistic.
- You can prototype and discard more ideas safely.
- You can spend more time on design and quality.

The net effect is **3–5× more validated “feature slices”**, **not** 10× more random LOC.

---

## 6. Model & Tool Routing (Specific to Throughput)

This complements the system-wide routing, but focused on dev speed:

- **Claude Max**
  - System-level planning, slicing features, architecture decisions.
  - Writing/refining specs and decision docs.

- **Claude Code**
  - Deep framework-specific changes in `/frontend`, `/backend`, `/agents`.
  - Non-trivial refactors, complex bug fixing.
  - Invoked primarily via Kilo Code or Cursor.

- **ChatGPT code models**
  - High-volume, low-risk code generation:
    - CRUD endpoints, forms, input validation, DTOs.
    - Client SDKs, type mappers, fixtures.
  - Invoked via Kilo Code or Cursor when you want fast scaffolding.

- **Local LLaMA/Qwen**
  - Offline summarisation of large files/logs.
  - Experimental refactors and code transformations on branches.

- **Cursor**
  - Default environment for reading code, quick edits, and inline suggestions.

- **Kilo Code (inside Cursor)**
  - “Do the whole multi-step job”:
    - Plan → edit multiple files → run tests → iterate.
  - Best for tasks that touch many files or require repeated trial/fix cycles.

---

## 7. Weekly Planning Heuristics

Assume you can sustain **4 feature-focused days + 1 refactor/ops day** per week.

That yields:

- **4 × (200–800) net LoC/week**  
  → **800–3,200 net LoC/week** merged.
- Plus a refactor day that might be 0 or negative net LoC but improves the system.

Approximate feature sizing:

- Small slice: 100–300 LoC, **0.5 day**.
- Medium slice: 300–800 LoC, **1–2 days**.
- Large slice: 800–1,500+ LoC → **must be split** into smaller slices.

Plan the week in terms of **“slices”**, not raw lines.

---

## 8. Measuring Actual Throughput

To replace estimates with hard data:

1. For each merged PR, log:
   - LoC added/removed (from `git diff` or GitHub).
   - Label: `feature`, `refactor`, `chore`, `bugfix`.
2. Weekly, review:
   - Avg net LoC/day for `feature` PRs.
   - How many slices were shipped.
3. Adjust:
   - If quality is poor or bugs are high → aim for *fewer* slices with more tests.
   - If CI is green and bug rate is low → consider slightly increasing slice throughput.

You do **not** optimise for maximum LoC; you optimise for **minimal LoC per unit of shipped value**, subject to maintaining the 200–800/day range as a sanity check.

---

## 9. Ground Rules

- No raw “AI code dumps” to main:
  - Every change goes through a PR and CI.
  - Every change maps to at least one feature/spec note and issue.

- Negative LoC is allowed and welcome:
  - Deletion of dead code and simplification is “progress”.

- If you regularly exceed 800–1,000 net LoC/day:
  - Assume quality is slipping and re-evaluate velocity vs. quality.

---

## 10. Prompt Library (Throughput-Focused)

See `30_KB/skills/skill-ai-coding-loop.md` for detailed prompt templates and workflows.

The short versions:

- Planning:
  - “Help me break this feature into slices that fit 200–800 net LoC/day.”
- Execution:
  - “Given this slice, produce a file-by-file task list for Kilo Code/ Cursor.”
- Reflection:
  - “Given these PRs, estimate my effective slices and net LoC/day for the week.”
