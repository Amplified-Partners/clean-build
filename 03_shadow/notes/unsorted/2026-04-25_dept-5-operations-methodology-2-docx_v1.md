## **Biological Decision Logic Taxonomy**

**Source / Creator:** Originated by Ewan Bramley (voice insight), formalised by Claude

### **Description**

A taxonomy of 7 biological logic patterns used to route decisions to the correct logic type. Each biological logic pattern maps to an Amplified Operational Protocol Law, creating a decision-routing system grounded in natural intelligence patterns.

The 7 patterns are: 1\. Bacterial / Quorum Logic (P.\>.3.i) — No leader. Each node senses locally. When enough nodes detect the same signal (quorum), action fires. Used for threat detection and death spiral early warning. 2\. Viral Logic (P.+.1.i) — Minimum viable payload. Find existing infrastructure, inject, replicate using host resources. Used for go-to-market, lead magnets, financial autopsy hooks. 3\. Slime Mold Logic (R.?.4.v) — No brain. Explores ALL possible paths simultaneously. Reinforces paths that find resources. Used for PUDDING mixing, cross-domain discovery, innovation. 4\. Octopus Logic (P.=.3.i) — Each arm has its own neural cluster. Central brain sets intent but does NOT micromanage. Used for multi-agent task execution with local autonomy and global coherence. 5\. Mycelial Logic (R.+.5.l) — Connects distant, unrelated nodes. Transfers resources from abundance to scarcity. Used for knowledge graphs and cross-client intelligence (anonymised). 6\. Human Logic (P.\~.1.m) — Single decision-maker. Oscillates between rational analysis and emotional intuition. Used for governance, ethics, mission changes, the Ulysses Clause. Some decisions MUST stay human. 7\. Silicon AI Logic (P.=.0.i) — Scale-free, instant, no emotion. Pure pattern recognition. Used for hybrid search, reranking, RAG pipeline.

### **Amplified Usage**

The taxonomy serves as the operational decision router for Amplified Partners. Every decision type is classified by its characteristic and routed to the optimal biological logic pattern. The Biological Logic → Operational Law Mapping links each pattern to one of the Eight Laws: Mycelial \= Law 1 (GitHub is the mycelium); Slime mold \= Law 2 (marking which paths found resources); Bacterial/Quorum \= Law 3 (binary threshold); Human \= Law 4 (metacognition); Silicon AI \= Law 5 (pattern storage); Octopus \= Law 6 (think locally first); Quorum \= Law 7 (every contributor is a signal); Viral \= Law 8 (a win replicates through the system). The Decision Routing Table is fully documented in SKILL-16 (METHODOLOGY 47).

## **Declarative Communication Protocol (70/30 Rule)**

**Source / Creator:** Amplified Partners — derived from PromptForge philosophy

### **Description**

The principle of spending 70% of effort on problem definition (comprehensive specs, success criteria, test cases up front) and only 30% on execution. Described as "exactly the PromptForge philosophy applied to code," this protocol produces higher-quality first attempts and fewer costly revisions.

Process steps: 1\. Spend 70% of effort writing comprehensive specification: what does success look like? what are the acceptance criteria? what are the failure modes? 2\. Write test cases before writing any code (Test-First / Agent-Implements method). 3\. Only then: give the spec to the agent for execution (30% of effort). 4\. The system does the prompt engineering, not the human.

### **Amplified Usage**

Applied to every coding and build task within Operations & Foundation. This protocol is listed as "Technique \#1 — What Actually Works at Scale" from optimal-ai-workflow research. It ensures that operational processes are thoroughly specified before implementation begins, reducing rework cycles and improving first-pass quality across all automation efforts.

## **Gap Analysis & Finite Lenses Framework (v1.0)**

**Source / Creator:** Amplified Partners — built over 31 hours, March 2026

### **Description**

A framework identifying 10 gaps and defining 12 Analytical Lenses for systematically assessing a business. Uses APQC process codes in the atom ID format (AMP-{APQC\#}-{SEQ}-{VERSION}). Every situation gets 1 primary \+ 1–2 secondary lenses. Each lens adjusts the PRS formula weights.

The 12 Analytical Lenses: Financial Health, Operational Efficiency, Customer Value, Growth & Market, People & Culture, Risk & Compliance, Technology & Infrastructure, Knowledge & Learning, Brand & Reputation, Innovation & R\&D, Governance & Ethics, Integration & Coherence.

Process steps: 1\. Apply the 12 Analytical Lenses to the client situation. 2\. Select 1 primary lens (highest relevance) \+ 1–2 secondary lenses. 3\. Adjust PRS (Process Rating System) formula weights according to selected lenses. 4\. Assign atom IDs to all processes using the format AMP-{APQC\#}-{SEQ}-{VERSION}. 5\. Apply 5 simultaneous taxonomy layers per atom: APQC, PUDDING, Bio-Logic, Semantic Dimensions, and Integration. 6\. Identify 10 gap categories and score against each. 7\. Use Three-Data-Pool structure: Living Data (current), Backup Data (historical), Testing Data (synthetic).

### **Amplified Usage**

This is the primary analytical engine for operational assessment. When onboarding a new client or reviewing an existing operation, the 12 lenses provide structured, multi-dimensional analysis. The atom ID system (AMP-{APQC\#}-{SEQ}-{VERSION}) ensures every process is uniquely identifiable and version-controlled. The framework integrates with the 360° OS for comprehensive gap identification.

## **Kaizen Fine-Tuning Loop**

**Source / Creator:** Amplified Partners — derived from Kaizen methodology

### **Description**

A weekly cycle combining Kaizen methodology with LoRA fine-tuning to continuously improve local models. Creates a compounding improvement flywheel: better local models → cleaner context → cheaper API calls → more fine-tuning budget → even better models.

Process steps: 1\. Curate: Weekly review of agent outputs; select high-quality examples for training. 2\. Train: Apply LoRA fine-tuning to the relevant local model using curated examples. 3\. A/B test: Run new model version against current model on 705-case golden dataset. 4\. Deploy: If new version outperforms on golden dataset → deploy; if not → analyse and repeat.

7 Amplified-specific fine-tuning datasets: Enforcer reports, Linear issues, build logs, monologues, marketing data, code reviews, rubric scores.

The operational engine for continuous model improvement. "The model improves every single week." This loop directly reduces API costs while improving output quality — a key operational efficiency driver.

## **PCO — Progressive Context Optimizer**

**Source / Creator:** Amplified Partners — designed March 2026

### **Description**

Amplified Partners' entry into the context optimisation space. A quality-verified context compression system shipped as a single Docker container, exposed as both an MCP server and an OpenAI-compatible HTTP proxy. Four unique differentiators vs. all open-source alternatives.

Process steps: 1\. Client points LLM API URL to PCO endpoint instead of direct LLM endpoint. 2\. PCO intercepts the request transparently (Transparent Proxy / Zero-Change Deployment). 3\. Dual Classifier with Veto: (a) Compressor classifier performs compression — summarising, extracting atomic facts, reducing tokens; (b) Quality Verifier checks if compressed output preserves required information — with power to VETO a bad compression and force re-compression. 4\. Compressed, quality-verified context forwarded to actual LLM endpoint. 5\. LLM response returned to client application (transparently). 6\. Kaizen Loop: Track which compressed facts the LLM actually references; feed signal back to improve future compressions. 7\. Rubrik-Driven Configuration: Non-technical users tune compression via JSON rubrik file.

Hot/Cold Tier Pattern: PCO manages active window (hot); Mem0 manages cold storage; PCO offloads to Mem0 when window fills.

### **Amplified Usage**

PCO is described as "genuinely novel" — no other open-source tool implements a quality gate on compression. The ecosystem document "The Mycelial Network" maps 87 ecosystem tools around PCO as the central node. It achieves 30–60% token reduction while maintaining quality through the Dual Classifier with Veto pattern. Non-technical users can tune compression behaviour through the Rubrik-Driven Configuration system.

## **PromptForge System**

**Source / Creator:** Amplified Partners — created March 2026

### **Description**

A 5-stage voice-first, model-adaptive, self-improving prompt engineering system. Converts raw voice input into model-specific optimised prompts with database-backed Kaizen improvement loops. Five design principles: voice-first, model-adaptive, data-driven, self-improving, minimal friction.

Stages: 1\. Capture: Whisper STT transcription \+ filler removal \+ Intent Extraction Protocol (5-dimension: primary task, context, constraints, desired output type, domain). Stores in PostgreSQL raw\_captures table. Low confidence (\<0.7) triggers interactive clarification. 2\. Refine: Meta-prompting engine generates model-agnostic canonical prompt. Self-evaluation loop with up to 3 refinement iterations. 3\. Model-Adapt: Claude/Grok/Perplexity-specific adapters with PromptBridge calibration for cross-model transfer (10–30% drift prevention). 4\. Execute and Score: LLM-as-judge \+ 6-metric composite score stored in PostgreSQL. 5\. Kaizen Engine: Pattern analysis across 1,000+ prompts; failure analysis via TextGrad; adapter evolution monthly; few-shot library growth (\>8.0 score prompts added); Prompt Drift Detection with re-calibration on model updates.

### **Amplified Usage**

PromptForge is the operational interface between human intent and AI execution. It eliminates the need for users to learn prompt engineering — the system handles it. Listed as LOST at extraction date (exists only in Perplexity session), with a noted bootstrap paradox: PromptForge needs PostgreSQL → needs Beast → needs rebuild. Recovery and rebuild is a high-priority operational task.

## **Three-Layer Code Audit Process**

**Source / Creator:** Amplified Partners — synthesised from industry tools

### **Description**

A three-layer code audit framework for evaluating and classifying existing code as KEEP / REFACTOR / REBUILD. Maps onto the existing CODE-TAXONOMY-AND-KAIZEN-v1 spec.

Layers: 1\. Layer 1 — Static Analysis: SonarQube Community Edition (30+ languages, quality gates, code smells, security vulnerabilities) \+ Pylint/Flake8/MyPy/Bandit for Python. 2\. Layer 2 — AI-Powered Deep Review: Claude Audit 7-dimension (security, code quality, performance, dependencies, testing, documentation, architecture — each 0–100) \+ CodeRabbit for PR review. 3\. Layer 3 — Architectural Review: Claude Opus via API with Curator Gate Specification; PRS scoring — flag anything \<7.0 for rebuild.

Output classification per module: KEEP / REFACTOR / REBUILD.

### **Amplified Usage**

The systematic approach to technical debt management. Every code module receives a three-layer evaluation and is classified into one of three action categories. The PRS threshold of 7.0 provides an objective rebuild trigger. This process integrates with the broader Build Quality Framework and AMPS methodology to ensure code quality across the entire operational codebase.
