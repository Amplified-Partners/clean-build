### Kilo Code Excellence Framework

#### 1. Prompt Structure (Think → Plan → Execute → Review)

**ANALYZE FIRST (Think):**
```
Analyze the current state:
- What exists? (files, structure, dependencies)
- What's the goal? (specific, measurable outcome)
- What constraints? (tech stack, time, resources)
- What risks? (security, performance, maintainability)
```

**PLAN SECOND (Strategic Breakdown):**
```
Create implementation roadmap:
1. File structure (what goes where)
2. Dependency order (what depends on what)
3. Validation checkpoints (how to verify each step)
4. Rollback strategy (how to recover if broken)
```

**EXECUTE THIRD (Incremental Implementation):**
```
Implement one atomic unit at a time:
- Single file or function
- With tests/validation
- Verify before proceeding
- Document as you build
```

**REVIEW FOURTH (Quality Gates):**
```
Before marking complete:
✓ Code runs without errors
✓ Meets original requirements
✓ Follows style guidelines
✓ Documented for future maintenance
✓ No security vulnerabilities
```

## Part 2: World-Class Information Comprehension

### Deep Analysis Framework

#### 1. Multi-Layer Information Processing

**Surface Layer (Facts & Data):**
- What is explicitly stated?
- What are the key facts, figures, dates?
- What terminology is used?
- What sources are cited?

**Pattern Layer (Relationships & Structure):**
- How do concepts relate to each other?
- What patterns emerge across sources?
- Where are the contradictions or gaps?
- What frameworks or models are implied?

**Principle Layer (Underlying Rules):**
- What core principles drive the information?
- What assumptions are being made?
- What mental models are at play?
- What would an expert in this field recognize?

**Application Layer (Actionable Insights):**
- How does this apply to the user's context?
- What decisions does this inform?
- What actions should be taken?
- What risks or opportunities does this reveal?

#### 2. Expert-Level Source Evaluation

**Authority Assessment:**
- Is this a primary source or secondary?
- What's the author's credibility in this domain?
- Is this peer-reviewed or professionally vetted?
- What's the publication date and relevance?

**Bias Detection:**
- What's the author's perspective or agenda?
- What information might be missing or downplayed?
- Are alternative viewpoints represented?
- Is data presented selectively?

**Quality Markers:**
- Specific data vs. vague generalizations
- Clear methodology vs. unsupported claims
- Nuanced analysis vs. oversimplification
- Transparent limitations vs. overconfidence

#### 3. Synthesis Protocol

When analyzing multiple sources:

**Step 1: Categorize Information**
- Group by theme, concept, or domain
- Identify overlapping vs. unique insights
- Note consensus vs. debate areas

**Step 2: Cross-Validate Claims**
- Which facts appear across multiple sources?
- Where do sources disagree and why?
- What can be verified independently?

**Step 3: Build Mental Models**
- Create frameworks that explain the domain
- Identify cause-and-effect relationships
- Map decision trees or workflows

**Step 4: Extract Decision-Ready Insights**
- What are the 3-5 most important takeaways?
- What specific actions are recommended?
- What questions remain unanswered?

```cypher
// Link experts to principles

// Link decisions to principles used
(decision)-[:APPLIES]->(principle)

// Link decisions to outcomes
(decision)-[:LED_TO]->(outcome)

// Link insights to evidence
(insight)-[:SUPPORTED_BY]->(evidence)

// Link projects to decisions
(project)-[:CONTAINS]->(decision)
```

**Query Patterns:**

"Show me how I've applied Kennedy principles in past client work:"
```cypher
MATCH (e:Expert {name: "Dan Kennedy"})<-[:CREATED_BY]-(p:Principle)
MATCH (d:Decision)-[:APPLIES]->(p)
MATCH (d)-[:LED_TO]->(o:Outcome)
WHERE o.success = true
RETURN p.name, d.context, o.result, d.date
ORDER BY d.date DESC
LIMIT 10
```

"Which principles lead to highest client retention?"
```cypher
MATCH (p:Principle)<-[:APPLIES]-(d:Decision)
MATCH (d)-[:LED_TO]->(o:Outcome)
WHERE o.metric = "retention" AND o.value > 0.8
RETURN p.name, p.expert, COUNT(d) as usage_count, AVG(o.value) as avg_success
ORDER BY avg_success DESC
```

**Naming Conventions:**

**Files & Directories:**
- Use lowercase with hyphens: `user-service/`, `data-processor.py`
- Be descriptive: `authentication.py` not `auth.py` (unless ubiquitous)
- Group by domain, not type: `/users/` not `/models/user.py` + `/services/user.py`

**Variables & Functions:**
- snake_case for Python: `calculate_total_cost()`
- camelCase for JavaScript: `calculateTotalCost()`
- SCREAMING_SNAKE_CASE for constants: `MAX_RETRIES = 3`

**Classes:**
- PascalCase: `UserAuthenticationService`
- Noun-based: `PaymentProcessor` not `ProcessPayment`

**Git Branches:**
- `feature/add-payment-processing`
- `bugfix/fix-login-timeout`
- `hotfix/patch-security-vulnerability`

**Git Commits:**
```
type(scope): Short description (50 chars max)

Detailed explanation if needed (72 chars per line max).
Explain WHAT changed and WHY, not HOW (code shows how).

- Bullet points okay
- Reference issues: Fixes #123
```
