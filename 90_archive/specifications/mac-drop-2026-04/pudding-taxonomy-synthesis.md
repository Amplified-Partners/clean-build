## 1. ORIGIN AND ATTRIBUTION

**Theoretical foundation:** Don R. Swanson (1924–2012), Literature-Based Discovery (LBD), 1986.  
**Original paper:** "Fish oil, Raynaud's syndrome, and undiscovered public knowledge." *Perspectives in Biology and Medicine*, 30(1), 7–18.  
**Core insight:** Knowledge that has never been in the same room can be connected — if you find the bridge.

**Adapted by:** Ewan Bramley + Claude (AI) — for business knowledge synthesis, not biomedical literature.  
**System named:** The Pudding. British tradition of combining disparate ingredients into something greater than the sum of parts.

### The Swanson ABC Model

```
A → B  (known connection, one field)
B → C  (known connection, different field)
∴ A → C  (undiscovered connection — the pudding)
```

**Swanson's original example:**
- Fish oil → reduces blood viscosity (A→B, nutrition literature)
- Blood viscosity → causes Raynaud's disease (B→C, medical literature)
- Fish oil → treats Raynaud's disease (A→C, never previously connected)

**The Amplified Partners adaptation:** Instead of biomedical papers, inputs are named expert frameworks, research domains, biological architectures, and any other knowledge sources. The critical adaptation is **neutral taxonomy labelling** — stripping expert names and domain labels to expose the underlying mechanisms that enable cross-domain bridging.

## 2. PUDDING 2026 LABEL NOTATION — COMPLETE

Every concept receives a 4-character structural label in the format: **`WHAT.HOW.SCALE.TIME`**

Total possible labels: **7 × 7 × 7 × 6 = 2,058** (canonical statement: "2,401 possible labels" appears in the Telegraph Pole — see gap note in Section 19)

### WHAT (Entity Type) — 7 codes

| Code | Type | Description |
|------|------|-------------|
| `P` | Process | An action, procedure, or workflow |
| `I` | Information | Pure data, knowledge, or content |
| `R` | Relation | A connection, link, or relationship between things |
| `E` | Entity | A thing, product, or artifact |
| `S` | State | A condition, status, or situation |
| `C` | Constraint | A rule, limit, or boundary |
| `M` | Meta | A model, framework, or description of descriptions |

### HOW (Dynamic Behaviour) — 7 codes

| Code | Behaviour | Description |
|------|-----------|-------------|
| `=` | Stable | Unchanging, consistent |
| `+` | Amplifying | Growing, compounding, increasing |
| `-` | Dampening | Shrinking, reducing, decaying |
| `>` | Tipping | Threshold-based, binary trigger |
| `~` | Oscillating | Cycling, recurring, rhythmic |
| `!` | Disrupting | Breaking patterns, forcing change |
| `?` | Emerging | New, uncertain, forming |

### SCALE (Scope of Effect) — 7 codes

| Code | Scale | Description |

| `4` | Network | Connected group, community |

### TIME (Duration / Persistence) — 6 codes

| Code | Duration | Description |

### How to Apply Labels (4-Step Method)

1. **Read** the concept and strip away domain name and expert attribution
2. Ask **"What TYPE of thing is this?"** → WHAT code
3. Ask **"How does it BEHAVE over time?"** → HOW code
4. Ask **"How WIDE is its effect?"** → SCALE code
5. Ask **"How LONG does it persist?"** → TIME code
6. Write the label: `WHAT.HOW.SCALE.TIME`
7. Look for other concepts with identical or near-identical labels across different domains — these are highest-value pudding candidates

## 4. THE STEP-BY-STEP TECHNIQUE

### STEP 1: Research Phase
- Run the topic through search (triple search preferred — Brave, Tavily, Exa; or Perplexity equivalent)
- **Always** run two queries: **success patterns AND failure patterns**
- Collect named techniques, not just general findings
- Aim for 4–7 techniques per domain/area
- Extract from vault files when working with existing knowledge

### STEP 2: Neutral Labelling (The Taxonomy)
- For each technique, assign: a **neutral label**, **semantic dimensions** (3–7 from the ontology), and **What/How/Why**
- **CRITICAL:** Do NOT group by expert or domain. Flatten everything into ONE list.

### STEP 3: Apply PUDDING 2026 Label
- Every concept gets a 4-character structural label: `WHAT.HOW.SCALE.TIME`

### STEP 4: Pattern Matching — Find the B-Terms
- Look across ALL techniques for shared dimensions and matching PUDDING labels
- Two techniques from different domains sharing a dimension = candidate bridge
- **Identical PUDDING labels from different domains are the highest-value signal**

### STEP 5: ABC Connection — Build the Recipe
- Name the A (domain 1 technique)
- Name the B (shared bridge mechanism)
- Name the C (domain 2 technique)
- State the hypothesis: what does A→C reveal that neither A nor C revealed alone?
- **The test:** Is the A→C connection non-obvious? If yes, it's pudding. If no, it's a summary.

### STEP 6: Score the Recipe
- See Section 6 for all three scoring formulas

### STEP 7: State the 1+1=3 Insight
- Name the C explicitly — the new understanding that didn't exist before
- If you can't name it, you haven't made pudding — you've made a list
- Both experts (A and B) must say "I never thought of it that way"

### STEP 8: Validate or Label as Hypothesis
- If tested: mark `tested_internal` / `tested_client` / `proven`
- If not tested: mark `hypothesis`
- Every recipe must have a **testable prediction**

## 6. SCORING FORMULAS — ALL THREE

### Formula 1: Simple Recipe Score (for any two-source connection)
```
Recipe Score = (Shared Dimensions × 2) + Unique_A + Unique_B
```
- **Minimum viable recipe:** Score ≥ 5 (at least 1 shared dimension, at least 1 unique from each source)
- **High-value recipe:** Score ≥ 8 (cross-domain, different mechanism types)

### Formula 2: Advanced Cross-Domain Mix Score
```
Score = (Domain Distance × Pattern Alignment) + Gap Complement + Tension Bonus
```
Where:
- **Domain Distance** (1–6): How far apart are the source domains?
- **Pattern Alignment** (1–10): How closely do the PUDDING labels match?
- **Gap Complement** (1–5): Does the combination fill a gap neither source fills alone?
- **Tension Bonus** (0–3): Is there productive tension between the approaches?

**Thresholds:**
- Score ≥ 13: Worth building
- Score ≥ 18: Build immediately
- Score ≥ 20: Highest priority

### Formula 3: Automated Mixer Formula (from Mathematical Validation)
```
Pudding Rank = (Jaccard_slot × Domain_Distance) + (Unique_dimensions / Total_dimensions)
```
**Filters to apply before ranking:**
- Jaccard_slot ≥ 0.75 (3/4 or 4/4 label match)
- Domain_Distance ≥ 3/6 (at least medium domain separation)
- Individual rubric score ≥ 12/20 for both concepts

## 9. BIOLOGICAL DECISION LOGIC TAXONOMY (ALL 7 LOGICS)

**Source:** `PUDDING-MIX-002-biological-decision-logic-2026-03-10.md`  
**Origin:** Ewan Bramley (voice insight identifying biological decision architectures as an agent design pattern), Claude Opus (formalisation, PUDDING labelling, decision routing table)

### 1. Bacterial / Quorum Logic — `P.>.3.i`
**Architecture:** Threshold-based, local sensing  
**How it decides:** No leader. Each node senses local conditions independently. When enough nodes detect the same signal (quorum), action fires. Binary: below threshold = nothing. Above threshold = coordinated response.  
**Evolved for:** Collective action without central coordination. Biofilm formation, bioluminescence, virulence factor expression.  
**Agent application:** Death Spiral Detection (Rubrik 12). Don't fire on one metric. Wait until enough independent signals converge — revenue down + footfall down + staff hours cut. Quorum = intervention.

### 2. Viral Logic — `P.+.1.i`
**Architecture:** Hijack and replicate  
**How it decides:** Minimum viable payload. Find existing infrastructure. Inject. Replicate using host resources. Spread to next host.  
**Evolved for:** Maximum distribution with minimum own resources.  
**Agent application:** Go-to-market / Financial Autopsy Hook. The Xero/Sage API connection is viral architecture — minimum payload (one API call), hijacks existing infrastructure (their accounting software), replicates value across the host's business, spreads via word-of-mouth.

### 3. Slime Mold Logic — `R.?.4.v`
**Architecture:** Decentralised pathfinding  
**How it decides:** No brain. Explores ALL possible paths simultaneously. Reinforces paths that find resources. Starves paths that don't. Emergent optimisation without any central planner. Famously replicates Tokyo rail network.  
**Evolved for:** Finding the shortest path between food sources.  
**Agent application:** PUDDING Mixer / Cross-Domain Discovery. Explore all possible concept combinations simultaneously. Score each. Reinforce high-scoring paths (build them). Starve low-scoring paths (ignore them). **The mixer IS a slime mold.**

### 4. Octopus Logic — `P.=.3.i`
**Architecture:** Distributed nervous system  
**How it decides:** Each arm has its own neural cluster and makes local decisions autonomously. Central brain sets intent and general direction but does NOT micromanage. Arms can disagree and still function.  
**Evolved for:** Complex manipulation in unpredictable environments.  
**Agent application:** Agent Triad (Filter/Gatekeeper/Strategist). Each agent makes local decisions within its domain. The Orchestrator (central brain) sets intent — "help this customer" — but doesn't dictate how the Filter filters or how the Strategist strategises. Local autonomy, global coherence.

### 5. Mycelial Logic — `R.+.5.l`
**Architecture:** Resource network / underground connections  
**How it decides:** Connects distant, unrelated nodes. Transfers resources (nutrients, signals, warnings) from abundance to scarcity. The "wood wide web" — trees share resources through fungal networks.  
**Evolved for:** Ecosystem-scale resource optimisation. Cross-species mutual aid.  
**Agent application:** Knowledge Graph (FalkorDB / Graphiti). Underground connections between distant knowledge domains. Transfer insights from one client's abundance to another's scarcity (anonymised). The graph IS a mycelial network.

### 6. Human Logic — `P.~.1.m`
**Architecture:** Centralised executive + emotional weighting  
**How it decides:** Single decision-maker (prefrontal cortex). Oscillates between rational analysis and emotional intuition. Slow but deeply contextual. Incorporates narrative, values, identity, and social consequences.  
**Evolved for:** Complex social decisions with long-term consequences. Trust, alliance, reputation.  
**Agent application:** Founder decisions / Immutable Foundation Code. Some decisions MUST stay human — governance, ethics, mission changes, the Ulysses Clause. The system presents data using all other logics, but final call on certain classes stays with Ewan. Oscillation between reason and gut feel is a FEATURE, not a bug.

### 7. Silicon AI Logic — `P.=.0.i`
**Architecture:** Statistical pattern matching  
**How it decides:** Scale-free, instant, no emotion. Pure pattern recognition against training distribution. Confidence is statistical, not experiential.  
**Engineered for:** Processing scale and speed that biological systems cannot achieve.  
**Agent application:** Hybrid Search, Cross-Encoder Reranking, RAG pipeline. Fast, reliable, no ego. But it doesn't know what it doesn't know — which is why it needs the other logics to ROUTE decisions to it appropriately.

## 17. WHAT MAKES A GOOD PUDDING / WHAT KILLS IT

### High-Confidence Signals

1. **Cross-domain is essential.** Connections within the same domain are not pudding — they're just good reading. Pudding requires at least two different domains or expert traditions.

2. **The bridge must be neutral.** If you name the bridge after one of the domains ("the Dalio connection"), you've not found a bridge — you've found a one-way street. A true bridge has no owner.

3. **Non-obvious is the test.** Would an expert in domain A already know this? Would an expert in domain B already know this? If both would say "obviously" — it's not pudding. If both would say "I never thought of it that way" — it is.

4. **State the 1+1=3 explicitly.** The new understanding (C) must be named. "These two things are related" is not enough. "Combining these two creates a third thing that neither produced alone" is pudding.

5. **Every pudding is a hypothesis until tested.** Don't treat it as proven until it's been applied and validated.

6. **Identical PUDDING labels from different domains are the highest signal.** When two concepts from completely unrelated fields receive the same `WHAT.HOW.SCALE.TIME` code, that's structural equivalence worth investigating immediately.

### Failure Patterns — What Kills the Pudding

1. **Labelling by source, not mechanism.** If you call it "the Kennedy principle" you can only connect it to other Kennedy ideas. Strip the name. Label the mechanism.

2. **Summarising instead of connecting.** "These three techniques all agree that trust matters" is a summary. "These three techniques, when combined, reveal that trust is a prerequisite for urgency to work — without which urgency becomes coercion" is pudding.

3. **Stopping at two inputs.** The ABC model is minimum. The best pudding comes from A + B + C + D, where D is a mechanism from a fourth area that forces a constraint nobody expected.

4. **Collecting without flattening.** If you keep research in separate buckets (domain 1, domain 2, domain 3) and then look for connections, you'll find surface connections. If you flatten everything into one list of neutral mechanisms first, you find structural connections.

5. **Skipping the failure search.** The success patterns alone are not enough. Running the failure search — what consistently causes this to fail — and cross-referencing where both searches point to the same factor is where the highest-confidence signal lives.

6. **Using one decision logic for all decisions.** Different decision types need different biological architectures (MIX 002). Routing everything through one confidence gate is one-dimensional.
