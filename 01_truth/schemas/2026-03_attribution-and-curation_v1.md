## THE LAW

Every piece of data knows where it came from. Every database has a guardian. Every label is compact, honest, and machine-readable.

We are not that clever. We stand on the shoulders of giants — all the way back to amoeba. Every pattern we use came from somewhere. Every technique was figured out by someone before us. Every clever trick in this system has a lineage. We say so. Not because we are legally obliged to. Because it is honest. Because an honest audit trail is worth more than the illusion of originality. Because attribution is how we respect the people whose work we build on, and how we earn the right to have our own work built upon in return.

This document establishes three interlocking principles that govern how data is labelled, stored, accessed, and traced across all Amplified Partners systems:

1. **Attribution Throughout** — Every data object carries a record of its origins, the transformations it has undergone, and the agents that touched it. No exceptions.
2. **Curator Agents** — Every database has exactly one designated Curator Agent. No other process reads from or writes to that database directly. The Curator is the gatekeeper, the quality controller, the auditor, and the translator.
3. **Compressed Label Codes** — Labels in graph databases, RAG embeddings, and pipeline outputs use a structured coding system. Full-text labels are prohibited in storage. Codes are compact, machine-readable, and hierarchically meaningful.

These three principles are not independent features to be adopted selectively. They are a system. Attribution requires Curators to enforce it. Curators require compressed codes to make attribution manageable at scale. Compressed codes require attribution to give them meaning. Remove any one leg and the structure falls.

**Mathematics decides. Not vibes. Not urgency. Not "we'll add attribution later."**

This document is constitutional. Deviations require a formal amendment, not a workaround.

## SECTION 1: ATTRIBUTION — STANDING ON SHOULDERS

---

### 1.1 The Philosophy

Attribution is not legal compliance. It is not IP management. It is not bureaucratic overhead.

Attribution is honesty.

When we use a code pattern from a library, we name the library. When we cite research in an analysis, we cite the source. When we implement an algorithm described in a paper, we reference the paper. When we build on open-source work, we acknowledge the authors. When our work is used by someone else, our name is on it.

This is not because we are paranoid about credit. It is because the alternative — presenting work as though it sprang fully formed from our own minds — is a form of dishonesty, and dishonesty corrodes trust. A system that cannot trace its reasoning is a system that cannot be trusted. A system that cannot be trusted is a liability.

There is also a practical argument. Attribution creates auditable lineage. When a finding in a report turns out to be wrong, the attribution chain tells us where the error entered. When a model produces an unexpected result, the attribution chain tells us which training data influenced it. When a client asks "how did you arrive at this conclusion?", the attribution chain answers the question.

The PROV-O model from the W3C provides the formal vocabulary we use: an **Entity** (a thing — a document, a finding, a node) was generated by an **Activity** (a process — a search, a transform, an analysis) which was associated with an **Agent** (who did it — a model, a pipeline, a human). These three concepts are sufficient to describe any provenance relationship in our system.

We are not the first to get this right. Medical informatics, academic publishing, and legal documentation have required rigorous provenance tracking for decades. We inherit their discipline.

---

### 1.2 Attribution by Context

Attribution is not one-size-fits-all. The same principle — "everything knows where it came from" — applies differently depending on what "it" is and what "came from" means in that context.

---

#### Research Attribution

When the system performs research — web searches, document analysis, paper ingestion, Pudding technique cross-domain discovery — every finding carries a provenance record.

A finding is an **Entity**. The search or analysis that produced it is an **Activity**. The agent that executed the activity is an **Agent**.

Every research finding must carry:

- **Source identifier** — URL or DOI for external sources; vault path for internal documents
- **Author or publication** — Where known and resolvable
- **Date retrieved** — The date the source was accessed, not the date it was published
- **Confidence level** — One of three values: `primary` (direct source, first-hand), `secondary` (summarised or interpreted from a primary source), `inferred` (derived by reasoning across multiple sources without a single authoritative citation)
- **Agent identifier** — Which AI model, pipeline step, or human researcher produced the finding

Confidence levels are not aspirational. If the system cannot trace a finding to a primary source, it must be marked `inferred`. Marking inferred findings as primary is a validation failure.

---

#### Pipeline Attribution

When data flows through a pipeline — vault ingest, naming validation, Pudding extraction, entity resolution, embedding creation — every output carries a transform chain.

The transform chain answers: "What was done to this data, in what order, by what agents, and did each step pass its gates?"

Every pipeline output must carry:

- **Input hash** — Content-hash of the data before transformation began. This is the deduplication anchor.
- **Transform chain** — Ordered list of operations applied, each with the agent that executed it and the gate result at that checkpoint
- **Timestamps** — ISO 8601 timestamp at each stage; not just start and end, but each gate
- **Gate results** — Pass or fail for each gate the data passed through, with the gate identifier from the Validation Methodology

The transform chain is not optional even for "simple" pipelines. A pipeline with one step still has a transform chain with one entry.

---

#### Batch Validation Attribution

When the system processes a batch — validating 100 vault files against the naming convention, checking a corpus of documents for schema compliance, running quality gates across an overnight ingest — attribution must be tracked at three levels simultaneously.

**Per-file level:**
- Which rules fired
- Which rules passed
- Which rules failed
- Which Curator performed the validation
- Timestamp of validation

**Per-rule level:**
- Hit rate across the batch — what percentage of files triggered this rule
- Error type distribution — are failures concentrated in specific error patterns
- New errors — failures that match no known pattern in the Error Anticipation Registry (these are escalated to Kaizen)

**Per-batch level:**
- Aggregate pass rate and fail rate
- Statistical confidence on the rates (not just a raw percentage — a batch of 10 and a batch of 10,000 have very different certainty)
- Anomaly flags — files that fail in unusual ways, or fail a combination of rules that has not been seen before, are flagged for human review before any automated remediation is applied

This three-level tracking is what makes the Error Anticipation Registry useful over time. Without it, we have a log of failures. With it, we have intelligence about failure patterns.

---

### 1.3 The Attribution Record Format

All data objects carry a standard attribution record. The Curator stamps this record when it writes the object. The record is immutable once written — corrections are additive, not overwriting.

The canonical format is YAML, stored as a structured property on the data object:

```yaml
attribution:
  created_by:
    agent: "pudding-extract-v2"        # Which agent or process created this
    human: "ewan-bramley"              # Human responsible (if applicable)
    model: "claude-sonnet-4"           # LLM model used (if applicable)
  derived_from:
    - source: "https://doi.org/10.1234/example"
      type: "academic-paper"
      retrieved: "2026-03-15"
      confidence: "primary"
    - source: "vault://research/market-analysis-2026-01-v1.md"
      type: "internal-document"
      confidence: "primary"
  transforms:
    - step: 1
      operation: "pudding-label-extract"
      agent: "curator-kg"
      timestamp: "2026-03-15T14:30:00Z"
      gate: "G1-pass"
    - step: 2
      operation: "entity-resolve"
      agent: "curator-kg"
      timestamp: "2026-03-15T14:30:05Z"
      gate: "G3-pass"
  licence: "CC-BY-4.0"                # If externally sourced; omit for internal
  amplified_version: "v1"
```

**Field rules:**

| Field | Required | Notes |
|-------|----------|-------|
| `created_by.agent` | Mandatory | The pipeline step or agent identifier. Never blank. |
| `created_by.human` | Conditional | Required if a human was the proximate creator or approver |
| `created_by.model` | Conditional | Required if an LLM generated or substantially transformed the content |
| `derived_from` | Mandatory | At least one entry required. If genuinely original, use `type: "original"` — do not omit the field |
| `derived_from[].confidence` | Mandatory | Must be one of: `primary`, `secondary`, `inferred` |
| `transforms` | Mandatory for pipeline outputs | Empty list is permitted only for raw ingested documents before any processing |
| `licence` | Conditional | Required for any externally sourced material |
| `amplified_version` | Mandatory | Schema version of this attribution record |

Attribution records are stored compressed in the graph database using the Curator's lookup mechanism. The record schema version (`amplified_version`) ensures forward compatibility when the record format evolves.

---

### 1.4 Attribution Enforcement

Attribution is enforced at the gate level. This is not a soft recommendation. These are hard gates.

**G1 — Structure Gate:** Every data object entering any Amplified database MUST have an attribution record. An object without an attribution record fails G1 and is rejected. The Curator does not write it. The Curator does not queue it for later. The Curator rejects it and logs the rejection.

**G2 — Completeness Gate:** The attribution record MUST have at minimum `created_by` and `derived_from` populated with valid values. A record with empty or null values in these fields fails G2 regardless of whether the fields are present.

**G6 — Cross-System Gate:** When data crosses a system boundary — from FalkorDB into a report, from the vector store into a prompt, from the document store into a client deliverable — the attribution record MUST transfer with it. Attribution does not stop at the database boundary. The client deliverable is a data object. It carries attribution.

A data object that enters the system with full attribution and exits without it has had its attribution stripped. Stripping attribution is a validation failure equivalent in severity to data corruption.

## SECTION 2: CURATOR AGENTS — THE DATABASE GUARDIANS

---

### 2.1 The Principle

No agent, no pipeline, no automated process, and no human writes directly to a database. No exceptions.

All database interactions — reads and writes — go through the designated Curator Agent for that database. The Curator is the single point of quality control. The Curator is the single point of schema enforcement. The Curator is the single point of attribution tracking. The Curator is the only process that holds database credentials.

This is not a convenience pattern. It is a constitutional rule. It is also a security boundary. A system in which 15 different agents each hold a copy of the FalkorDB password is a system in which one compromised agent means 15 credential rotations, 15 audit log reviews, and 15 potential data corruption events. A system in which one Curator holds the credentials is a system in which one rotation covers everything.

---

### 2.2 Why This Matters

**Schema integrity.** When any agent can write to the database, schema drift is inevitable. Within weeks, nodes have inconsistent property sets, edge types are named differently across agents, and queries that relied on consistent structure begin to fail. When the Curator is the only writer, schema is enforced at a single point and drift is architecturally impossible.

**Attribution guarantee.** The Curator stamps attribution on every write. It is not possible to write a record without attribution because it is not possible to bypass the Curator. Attribution is not a discipline that requires everyone to remember. It is a structural guarantee.

**Compressed code translation.** The Curator maintains the code registry and translates between human-readable labels and compressed codes. If any agent could write to the database directly, it could write full-text labels and bypass the compression system. The Curator-only rule is what makes compressed codes enforceable.

**Audit trail.** Every read and every write passes through one gateway. Every operation is logged. The audit trail is complete by construction, not by convention.

**Error registry integration.** Before writing any record, the Curator validates it against the Error Anticipation Registry. A direct write bypasses this validation. Curator-mediated writes cannot.

**Key rotation.** Rotating database credentials requires updating one Curator. Without this pattern, it requires updating every agent, every pipeline, and every service that holds a credential.

---

### 2.3 The Curator Roster

One Curator per database. Permanent assignments. Changes to the Curator Roster require a formal amendment to this document.

| Curator Agent | Database | Primary Responsibilities |
|---------------|----------|--------------------------|
| `curator-kg` | FalkorDB (Knowledge Graph) | Node and edge creation, entity resolution, Pudding label management, temporal edge maintenance, code registry management |
| `curator-vec` | Qdrant / pgvector (Vector Store) | Embedding storage, similarity index maintenance, metadata enrichment, hybrid search coordination |
| `curator-doc` | PostgreSQL (Document Store) | Document metadata, version tracking, file registry, vault index maintenance |
| `curator-state` | Temporal (Workflow State) | Workflow history, checkpoint management, activity results, retry state |
| `curator-config` | YAML / Config Store | Configuration validation, secret rotation tracking, environment variable management |

Each Curator is a named, versioned Docker container on Beast. Each Curator has its own audit log. Each Curator exposes a typed API. None of them share credentials with each other or with any other service.

---

### 2.4 Curator Agent Architecture

Each Curator Agent exposes a typed API — a defined set of operations with defined inputs, outputs, and error codes. There is no raw database access through the Curator. A calling agent cannot pass arbitrary SQL or Cypher to the Curator and have it executed. The Curator's API is the complete surface area of allowed operations.

**Every Curator Agent:**

1. **Exposes a typed API** — Operations are named, typed, and versioned. `create_node(data, attribution)` is a valid call. `execute_query("MATCH (n) RETURN n")` is not.
2. **Validates every write against the gate sequence** — G1 through G7 in the order defined in `VALIDATION-METHODOLOGY-v2`. A write that fails any gate is rejected.
3. **Stamps attribution on every record** — The Curator adds its own identifier to the attribution chain on every write. Even if the calling agent provides incomplete attribution, the Curator's intervention is recorded.
4. **Translates labels to compressed codes** — The Curator accepts human-readable labels in the write request and stores compressed codes. It accepts compressed codes in read requests and returns human-readable labels. The translation is invisible to calling agents.
5. **Logs every operation** — Every read and write is written to the Curator's structured audit log: timestamp, operation, calling agent, outcome, and any gate failures.
6. **Rejects writes that fail validation** — Fail closed, not fail open. When a write fails validation, the Curator rejects it and returns an error. It does not attempt to fix the data and write it anyway. It does not queue it for later. It rejects it.
7. **Reports anomalies to Kaizen** — Failures that match no known pattern in the Error Anticipation Registry are escalated to the Kaizen Department in addition to being rejected.

**Write flow:**

```
Pipeline Agent
  → Calls curator-kg.create_node(data, attribution)
  → Curator validates attribution record (G1: structure present)
  → Curator validates attribution completeness (G2: fields populated)
  → Curator checks data against Error Anticipation Registry
  → Curator resolves entity against existing graph nodes (G3: deduplication)
  → Curator compresses labels to codes via code registry
  → Curator writes to FalkorDB with compressed codes
  → Curator stamps its own identifier to the attribution chain
  → Curator writes to audit log
  → Curator returns confirmation with compressed node ID and attribution hash
```

**Read flow:**

```
Pipeline Agent
  → Calls curator-kg.get_node(code_id, expand_labels=True)
  → Curator queries FalkorDB with compressed code
  → Curator expands code to human-readable label via code registry
  → Curator appends attribution record to the response
  → Curator logs the read
  → Curator returns data with expanded labels and full attribution
```

The calling agent does not need to know that compression is happening. The API abstraction handles it entirely.

---

### 2.5 The Curator-Only Access Rule

This is constitutional. No exceptions permitted. No workarounds tolerated. No "just this once" for a debugging session.

If an agent needs data from a database, it asks the Curator. If a pipeline needs to write a record, it tells the Curator. If a developer needs to inspect the graph during debugging, they call the Curator's diagnostic endpoint. The Curator is the only process with database credentials.

**Consequences of violation:**

Any direct database write that bypasses the Curator is, by definition, unattributed. It may introduce schema drift. It has not been validated against the Error Anticipation Registry. It has not been logged. It cannot be audited.

Discovery of a direct database write is treated as a critical incident. The affected records are quarantined until the Curator can validate and re-stamp them. The pipeline or agent that made the direct write is suspended until the access point is closed.

This is not punitive. It is structural. The Curator pattern only works if it is inviolable. A gate with an exception is not a gate.

## SECTION 3: COMPRESSED LABEL CODES

---

### 3.1 The Problem

Graph database nodes and properties carry semantic meaning. When you label a node with its full descriptive text, you get human readability in exchange for a set of costs that compound as the graph grows.

**Memory usage.** In FalkorDB, which is built on Redis, every property string is stored in memory. A node property `topic: "Docker container orchestration with health checks and restart policies for production deployment on multi-tenant infrastructure"` consumes substantially more memory than `topic_code: "TEC-INFR-001"`. Across millions of nodes, this difference is not marginal.

**Traversal performance.** Every Cypher query that filters on a property value loads that property string into the comparison. Longer strings mean slower comparisons and larger intermediate result sets.

**Token budget pressure in RAG systems.** When embedding metadata alongside content — which the 2026 Virginia Tech research confirms significantly improves retrieval quality — every token spent on a verbose label is a token not available for content. The token budget is a zero-sum constraint. Longer metadata means shorter content.

**Fragile string matching.** Exact-match queries on full-text labels break on punctuation differences, capitalisation inconsistencies, synonym variation, and typos. Codes are exact by design.

**Bloated Cypher queries.** A query matching on `topic: "Docker container orchestration..."` is harder to read, harder to version, and harder to compose than one matching on `topic_code: "TEC-INFR-001"`.

Ewan identified this problem correctly. The solution has already been validated at scale by a domain that has dealt with exactly this trade-off for decades.

---

### 3.2 The Medical Informatics Precedent

Healthcare informatics solved the structured coding problem in the 1980s and 1990s. The systems they built are now used across hundreds of millions of patient records globally. We inherit their solution.

**SNOMED CT** (Systematised Nomenclature of Medicine — Clinical Terms) contains approximately 350,000 medical concepts. Each concept has a numeric code. `73211009` means "Diabetes mellitus". The code is compact, globally unique, and sits within a formal hierarchy — the code's relationships to parent and child concepts are defined in the ontology, not in the code string itself. A system can reason about the hierarchy without parsing any text.

**ICD-10** (International Classification of Diseases, 10th revision) uses structured alphanumeric codes that encode meaning in their structure. `E11.9` means "Type 2 diabetes mellitus without complications". `E` identifies the chapter (Endocrine, nutritional and metabolic diseases). `11` identifies the specific condition (Type 2 diabetes mellitus). `.9` specifies the detail (without complications). The code is parseable — a system that knows the structure can infer the domain from the first character without any lookup.

**LOINC** (Logical Observation Identifiers Names and Codes) provides codes for laboratory tests and clinical observations. `2345-7` means "Glucose [Mass/volume] in Serum or Plasma". The code is compact and the description is verbose. The code travels with the data; the description is resolved on read.

The principle across all three systems is identical:

1. A short, structured code replaces a long descriptive string in storage and transport
2. The code-to-description mapping lives in a maintained lookup table (our equivalent is the code registry, maintained by the Curator)
3. The code encodes hierarchy in its structure, so a system can filter by domain without a full lookup
4. Deprecated concepts are never deleted — they are marked deprecated with a pointer to the replacement

We apply exactly this principle to the Amplified knowledge graph, adapted for our domain vocabulary and operational context.

---

### 3.3 The Amplified Coding System

**Structure:** `{DOMAIN}-{CATEGORY}-{SEQUENCE}`

**DOMAIN** (2–3 uppercase characters): The broad conceptual area. Determines the highest-level filter applicable to any query.

**CATEGORY** (2–4 uppercase characters): The specific category within the domain. Together with the domain, provides the namespace for the sequence.

**SEQUENCE** (3–4 zero-padded digits): Unique identifier within the domain-category namespace. Auto-incremented by the Curator. Never reused. Never reset.

**Full code examples:**

```
TEC-INFR-001  =  "Docker container orchestration"
TEC-INFR-002  =  "FalkorDB graph database configuration"
TEC-INFR-003  =  "Traefik reverse proxy and routing"
TEC-GRPH-001  =  "Knowledge graph entity resolution"
TEC-GRPH-002  =  "Temporal edge invalidation"
TEC-GRPH-003  =  "Graph traversal optimisation"
TEC-LLM-001   =  "LiteLLM model routing and fallback"
TEC-LLM-002   =  "Embedding model selection"
BIZ-STRT-001  =  "Market entry strategy"
BIZ-STRT-002  =  "Competitive positioning"
BIZ-CLNT-001  =  "Client onboarding process"
FIN-CASH-001  =  "Cash flow forecasting"
FIN-COST-001  =  "Infrastructure cost modelling"
MKT-CONT-001  =  "Content marketing pipeline"
MKT-RSCH-001  =  "Market research methodology"
OPS-PIPE-001  =  "Pipeline orchestration pattern"
OPS-VLDT-001  =  "Validation gate architecture"
LEG-COMP-001  =  "GDPR compliance framework"
PPL-HIRE-001  =  "Recruitment and hiring process"
```

The structure is parseable without a lookup. A query about technology infrastructure can pre-filter on `TEC-INFR-*` before performing any semantic search. A query about business strategy can pre-filter on `BIZ-STRT-*`. This pre-filtering is the foundation of the RAG optimisation strategy in Section 5.

---

### 3.4 How Codes Improve the System

**In FalkorDB:**

Instead of:
```
topic: "Docker container orchestration with health checks and restart policies for production deployment"
```

The Curator stores:
```
topic_code: "TEC-INFR-001"
```

The full description lives in the code registry, accessible via the Curator's `expand_code` operation. The graph is leaner, traversals are faster, and Cypher queries are readable.

**In RAG embeddings:**

When creating contextual chunks for vector storage, the Curator makes a deliberate choice based on the embedding goal:

- **For semantic richness:** Expand the code to its full description before embedding. The embedding captures the meaning, not the code string.
- **For compact metadata:** Store the code as a filterable metadata field alongside the embedding. Queries can filter on domain before invoking the semantic search.

The 2026 Virginia Tech research confirms that embedding metadata alongside content consistently improves retrieval quality. Compact codes make this feasible within token budget constraints. The Curator manages this trade-off.

**In Pudding labels:**

The Pudding Technique generates cross-domain labels on vault documents to enable serendipitous discovery. More labels per document means more potential connections. Compressed codes make more labels possible within the same token budget.

A document might carry:
```
labels: ["TEC-GRPH-001", "BIZ-STRT-002", "FIN-CASH-001", "OPS-PIPE-001"]
```

Four codes, approximately 60 characters. The equivalent in full text would exhaust the available label budget. The Pudding Technique's discovery power scales with the number of cross-domain labels. Compressed codes are a multiplier.

**In Cypher queries:**

```cypher
MATCH (n {topic_code: "TEC-INFR-001"}) RETURN n
```

is faster, more precise, and less error-prone than:

```cypher
MATCH (n {topic: "Docker container orchestration..."}) RETURN n
```

The compressed form does not break on typos. It does not vary with capitalisation. It does not require fuzzy matching to account for synonym variation. It either matches or it does not.

---

### 3.5 The Code Registry

The Curator maintains a code registry: a structured lookup table that maps every code to its full label, description, synonyms, provenance, and lifecycle status.

The registry is the authoritative source of truth for all codes. It is versioned. It is auditable. It is maintained exclusively by the Curator. No other process creates, modifies, or deprecates codes.

**Registry format:**

```yaml
# code-registry.yaml — maintained exclusively by curator-kg
schema_version: "v1"
last_updated: "2026-03-17"
updated_by: "curator-kg"

domains:
  TEC:
    name: "Technology"
    description: "Technology infrastructure, platforms, models, and engineering patterns"
    categories:
      INFR:
        name: "Infrastructure"
        description: "Physical and virtual infrastructure, containerisation, networking, orchestration"
        codes:
          "001":
            label: "Docker container orchestration"
            description: "Container orchestration including health checks, restart policies, networking, and volume management for production deployment"
            created: "2026-03-01"
            created_by: "ewan-bramley"
            status: "active"
            synonyms:
              - "container orchestration"
              - "docker compose"
              - "container management"
              - "docker swarm"
          "002":
            label: "FalkorDB graph database configuration"
            description: "Configuration and tuning of FalkorDB instances including memory allocation, index management, and query optimisation"
            created: "2026-03-01"
            created_by: "ewan-bramley"
            status: "active"
            synonyms:
              - "falkordb config"
              - "graph database setup"
              - "redis graph configuration"
```

**The `synonyms` field is critical.** When a calling agent sends a human-readable label that does not exactly match any existing code label, the Curator checks all synonyms before creating a new code. This prevents synonym proliferation — the situation where `TEC-INFR-045` = "Docker" and `TEC-INFR-046` = "docker compose" and `TEC-INFR-047` = "container management" exist as separate codes for the same concept.

**Synonym management is a Kaizen responsibility.** The monthly code registry audit includes a synonym gap analysis: labels that were submitted without matching any synonym and therefore triggered new code creation, where the new code might actually be a synonym for an existing code.

---

### 3.6 Code Lifecycle Rules

**Creation:** New codes are created only by the Curator. Creation is triggered only when no existing code label or synonym matches the incoming label within an acceptable similarity threshold. The Curator does not create new codes speculatively or in bulk. Codes are created on demand, when data requires them.

**Sequence:** Sequence numbers are auto-incremented by the Curator within each domain-category namespace. Sequence numbers are never manually assigned. Sequence numbers never reset. If the first 10 codes in a category are created and 3 are subsequently deprecated, the next new code is `011`, not `008`.

**Deprecation:** Codes are never deleted. A deprecated code has its `status` field set to `deprecated`, a `deprecated_by` field pointing to the replacement code, and a `deprecated_date` field. The deprecated code remains in the registry permanently. Historical data that used the deprecated code remains valid. New writes that reference the deprecated code are redirected by the Curator to the replacement code with a logged warning.

**Immutability:** A code's `label` and `description` fields are effectively immutable once the code is in active use. Corrections to descriptions are permitted if the correction does not change the meaning (e.g., fixing a typo). Changes that alter the meaning require creating a new code and deprecating the old one.

**Synonyms:** Synonyms are additive. New synonyms can be added at any time. Synonyms cannot be removed once added (removing a synonym would cause historical synonym-matched writes to be unresolvable). Synonym additions are a Kaizen operation.

---

### 3.7 The Token Budget Argument

The benefit of compressed codes in the RAG context is quantifiable. The numbers below use a standard 512-token embedding window, which is the default context length for many embedding models.

**Scenario: Pudding labels embedded alongside a content chunk**

A typical content chunk occupies approximately 400 tokens. This leaves 112 tokens for metadata, including labels.

| Labelling approach | Tokens per label | Labels with 112 token budget | Labels with 50 token budget |
|-------------------|-----------------|------------------------------|------------------------------|
| Full-text verbose | ~12 tokens | ~9 labels | ~4 labels |
| Full-text concise | ~6 tokens | ~18 labels | ~8 labels |
| Compressed code | ~3–4 tokens | ~28–37 labels | ~12–16 labels |

Compressed codes provide approximately 3× more labels per embedding than full-text verbose labels within the same token budget. For the Pudding Technique, where the discovery power scales directly with the number of cross-domain labels per document, this 3× multiplier translates directly to 3× more potential symbiotic connections per document.

This is not a minor optimisation. It is an architectural advantage.

When semantic richness in the embedding matters more than label count, the Curator expands codes to full descriptions before embedding. When label count matters more (Pudding-mode discovery), the Curator keeps them compressed. The Curator makes this choice per-operation based on the calling agent's declared purpose.

## SECTION 4: HOW ALL THREE WORK TOGETHER

---

The three principles — Attribution, Curator Agents, and Compressed Label Codes — are designed as a single integrated system. This section shows the complete flow from research to storage to retrieval.

---

### 4.1 The Integrated Flow

**Step 1 — Research agent finds a paper on retry patterns:**

```
Research agent executes a web search
  → Finds paper: "Exponential Backoff in Distributed Systems" (2025)
  → Attribution stamped immediately:
      source: "https://doi.org/10.1234/retry-patterns"
      type: "academic-paper"
      retrieved: "2026-03-17"
      confidence: "primary"
      agent: "research-pudding-v1"
```

The finding is an attributed object from the moment of creation. Attribution is not added later.

**Step 2 — Pudding agent labels the finding with codes:**

```
Pudding agent analyses the finding
  → Generates cross-domain label candidates
  → Submits candidates to curator-kg.resolve_labels(candidates)
  → Curator checks each candidate against code registry + synonyms
  → Curator returns codes: ["TEC-INFR-003", "TEC-GRPH-002", "OPS-PIPE-001"]
  → Compact codes, not full text — room remains for more labels
  → Pudding agent appends additional domain codes: ["BIZ-STRT-001"]
```

The Curator is involved in label assignment. No agent self-assigns codes. This ensures the code registry remains consistent.

**Step 3 — Curator-kg receives the labelled finding:**

```
curator-kg.create_node(finding_data, attribution_record)
  → G1: Attribution record present? Yes — pass
  → G2: created_by and derived_from populated? Yes — pass
  → Error Registry: Does this finding match any known error pattern? No — pass
  → G3: Does this entity already exist in the graph? Check hash — no match — new node
  → Compress: codes already compressed — verify against registry — pass
  → Write to FalkorDB: node created with compressed codes and attribution hash
  → Audit log: operation recorded with timestamp, calling agent, outcome
  → Return: node ID "TEC-GRPH-kg-000247", attribution hash "sha256:a3f..."
```

**Step 4 — Later, a RAG query about infrastructure patterns:**

```
Query agent: "What do we know about retry patterns in distributed systems?"
  → curator-vec.hybrid_search(query, domain_hint="TEC")
  → Curator translates query to embedding
  → Curator pre-filters vector store to TEC-* codes (dramatically reduces search space)
  → Curator runs semantic search within filtered subset
  → Top result: node "TEC-GRPH-kg-000247"
  → Curator expands codes to human-readable labels for the response
  → Curator appends attribution chain: source, date, confidence, pipeline history
  → Returns: finding with expanded labels and complete provenance
```

The attribution chain is intact from the original paper through to the query response. Every step is auditable.

### 4.2 System Invariants

Across all flows, these invariants hold without exception:

1. No data object enters the system without an attribution record
2. No database write happens without Curator mediation
3. No full-text label is stored in the database
4. Every compression can be reversed (via the code registry)
5. Every attribution chain is forward-traceable (from source to current state) and backward-traceable (from current state to source)
6. The Curator's audit log provides a complete record of every operation

These invariants are enforced structurally, not conventionally. The Curator makes them true by construction. They do not depend on agents remembering to follow them.

## SECTION 5: RAG OPTIMISATION FOR CODED LABELS

---

### 5.1 Embedding Strategy

The vector store contains two types of objects that require different embedding approaches: **content embeddings** (the substance of a document or finding) and **label embeddings** (the codes that categorise it). The Curator manages both.

**Content embeddings:** When creating an embedding for a vault document, research finding, or extracted entity, the Curator:
- Takes the content chunk (typically 300–400 tokens)
- Expands any compressed codes in the content to their full descriptions (for semantic richness)
- Appends the most important codes as metadata fields on the vector record
- Creates the embedding and stores it with the expanded text

**Label embeddings:** Codes are stored as filterable metadata fields on every vector record — not embedded into the content string. This separation means:
- A code taxonomy change does not require re-embedding all content
- Domain pre-filtering can be applied before semantic search runs
- The hybrid search can weight semantic similarity and code match independently

**At query time:**
1. The Curator translates the natural language query into relevant domain codes using the code registry (fuzzy match on labels and synonyms)
2. The Curator applies a domain pre-filter to the vector store based on the matched codes
3. Semantic search runs within the filtered subset
4. Results are ranked by a combined score: semantic similarity × relevance weight
5. The Curator expands all codes to human-readable form before returning results

---

### 5.2 Metadata-Enriched Embeddings

The dual-encoder approach separates the encoding of metadata (codes) from the encoding of content. Each dimension is handled independently.

**Metadata encoder:** Maps compressed codes to a structured representation that captures the hierarchical relationships between codes. `TEC-INFR-001` and `TEC-INFR-002` are closer to each other than `TEC-INFR-001` and `BIZ-STRT-001`.

**Content encoder:** Maps the substance of the document or finding to a semantic embedding. This is the standard dense retrieval embedding.

**Fusion:** At retrieval time, a learned or configured weighting combines the metadata representation and the content embedding into a single relevance score. The weighting can be adjusted per query type:
- Precise domain lookups: higher metadata weight
- Exploratory discovery queries (Pudding mode): equal or higher content weight

This architecture means code updates do not invalidate content embeddings, and content updates do not require re-encoding metadata. The two dimensions are decoupled.

---

### 5.3 Pre-filtering with Codes

Before the semantic search engine touches a single embedding, the Curator can reduce the search space using code-based pre-filtering. This is not a minor optimisation in a large knowledge graph — it is the difference between searching 100,000 nodes and searching 3,000.

**Domain pre-filtering:**
- Query about finance → filter to `FIN-*` codes → semantic search across ~5% of the graph
- Query about infrastructure → filter to `TEC-INFR-*` → semantic search across ~2% of the graph
- Query about market strategy → filter to `MKT-*` and `BIZ-STRT-*` → semantic search across ~8% of the graph

**Multi-domain queries:** When a query spans domains (e.g., "How does our infrastructure cost relate to our pricing strategy?"), the Curator generates a union of pre-filters: `TEC-INFR-* OR FIN-COST-* OR BIZ-STRT-*`. The semantic search runs across the union. This is still significantly smaller than the full graph.

**Pudding discovery queries:** When the intent is cross-domain discovery rather than precise retrieval, pre-filtering is relaxed. The Curator uses a broader filter or no filter, allowing the semantic similarity to surface unexpected connections. This is the mode in which the full value of cross-domain labelling is realised.

The pre-filter decision is made by the Curator based on the `search_mode` field in the query request: `precise`, `broad`, or `discovery`.

---

### 5.4 The Attribution-Retrieval Connection

Every vector record carries its attribution metadata. When a retrieval operation returns a result, the calling agent receives not just the content but its complete provenance:

- Where did this finding originate?
- What confidence level was assigned to the source?
- What transforms were applied between the source and this representation?
- When was this indexed?

This means that a report generated from retrieved findings can trace every claim to its source. The LLM that synthesises the report does not need to remember where it learned something — the retrieval system carries the answer.

Attribution at retrieval time is the mechanism by which the Pudding Technique's cross-domain connections remain auditable. A connection between a technology finding and a finance pattern is only valuable if we know whether the connection is between two primary-source findings or between two inferred conclusions. Attribution carries that distinction.

## SECTION 6: ENFORCEMENT RULES

---

These rules are constitutional. They do not bend for convenience, urgency, or the belief that "just this once" is harmless.

---

**Every data object MUST carry an attribution record. No exceptions.**
A data object without attribution is not a valid data object. It is a data object in an invalid state. The Curator treats it as such: rejected, quarantined, and logged.

**Every database write MUST go through the designated Curator Agent. No exceptions.**
Direct writes bypass attribution, schema validation, error registry checks, and the audit log simultaneously. A direct write is not a shortcut — it is four simultaneous failures.

**Every label MUST use the compressed coding system. Full-text labels are prohibited in database storage.**
The Curator enforces this on every write. An incoming write that contains full-text labels is not rejected — it is converted. The Curator resolves the full-text label to a code before writing. If no code exists, the Curator creates one. The storage layer never sees full text.

**The Curator is the ONLY process with database write credentials. This is a security boundary, not a convenience.**
Credentials are issued once, to the Curator container, and rotated through the Curator's `rotate_credentials` operation. No other service requests or holds database write credentials. Discovery of credential leakage to a non-Curator process is a P0 security incident.

**Attribution records are immutable once written. Corrections are additive.**
If an attribution record contains an error, the correction is written as a new version of the record with a `correction_of` field pointing to the original. The original is preserved. There is no overwrite. An audit trail that can be overwritten is not an audit trail.

**The code registry is maintained by the Curator and audited by the Kaizen Department.**
No human or pipeline creates, modifies, or deprecates codes except via the Curator's API. The Kaizen Department reviews the registry monthly for consistency, synonym gaps, and orphaned codes.

**Mathematics decides. Not vibes. Not urgency. Not "we'll add attribution later."**
"Later" is when attribution is most needed and least likely to be accurate. Attribution stamped at creation time costs almost nothing. Attribution reconstructed after the fact costs enormous effort and is inevitably incomplete. Do it at creation time or do not do it at all.

## SECTION 7: KAIZEN INTEGRATION

---

The Kaizen Department is the quality improvement engine for this system. The attribution, Curator, and code systems generate observable metrics. Kaizen reviews those metrics on a defined schedule and acts on them.

---

### 7.1 Monthly Audits

**Code Registry Audit**

Scope: Complete review of the code registry.

Checks performed:
- Duplicate detection: are there two or more codes with overlapping descriptions or synonyms that should be merged?
- Orphan detection: are there codes that have not been applied to any data object in the past 90 days? (Candidates for deprecation review, not automatic deprecation)
- Synonym gap analysis: review new code creation events from the past 30 days — were any created because the Curator failed to match a synonym that should have been in the registry?
- Hierarchy review: does the domain-category structure still reflect the actual vocabulary of the knowledge graph, or have new domains or categories been created ad hoc that require formal registration?

Output: A code registry health report, delivered to the Kaizen log. Proposed synonyms, deprecations, and new category registrations are submitted as amendments to this document.

---

**Attribution Completeness Audit**

Scope: Random sample of 10% of data objects created in the past 30 days.

Checks performed:
- Attribution record present (G1 compliance rate)
- Required fields populated (G2 compliance rate)
- Attribution chain traceable to a known source (can we follow the `derived_from` chain to a resolvable external source or a vault document?)
- Cross-boundary integrity (G6): for data objects that were used in client deliverables or exported reports, is the attribution chain intact in the exported form?

Target: 100% G1 compliance. 99%+ G2 compliance. Any G6 failures are critical findings.

---

**Curator Performance Audit**

Scope: Operational metrics from each Curator's audit log.

Metrics reviewed:
- Request volume and trend
- Rejection rate: what percentage of incoming writes were rejected, and for which gate failures?
- Anomaly rate: what percentage of rejections triggered a new Error Anticipation Registry entry?
- Response time distribution: P50, P95, P99 — are there latency spikes that suggest contention?
- Code creation rate: how many new codes were created this month? Is this accelerating (normal for a growing knowledge graph) or unexpectedly spiking (possible synonym gap)?

---

**Code Coverage Metric**

The single most important progress indicator for the compressed coding system.

Definition: The percentage of knowledge graph nodes that carry at least one compressed code in their label set, as opposed to any full-text label.

Target trajectory:
- Month 1 (post-implementation): 60%+ coverage (legacy nodes are not immediately migrated)
- Month 3: 85%+ coverage
- Month 6: 95%+ coverage
- Steady state: 99%+ coverage (the remaining 1% accounts for nodes in the process of being resolved by the Curator)

Coverage below trajectory is a Kaizen action item. Coverage is reported in the monthly Kaizen log.

---

### 7.2 The Kaizen Escalation Path

| Finding | Severity | Response |
|---------|----------|----------|
| Attribution record missing on a new object | Critical | Quarantine object, alert Curator team, root cause within 24 hours |
| Direct database write detected | Critical | P0 incident — quarantine affected records, revoke access, full audit |
| G2 compliance rate below 99% | High | Kaizen review within 5 business days, remediation plan required |
| Code registry synonym gap identified | Medium | Submit synonym additions to Curator within 10 business days |
| Code coverage below trajectory | Medium | Kaizen review, migration plan for legacy nodes |
| Curator response time P95 exceeding 500ms | Low | Performance investigation, Beast resource review |
| New Error Anticipation Registry entries | Informational | Log and incorporate into next training cycle |

---

### 7.3 SLA Commitments

Consistent with companion documents:

- Curator write operations: P50 < 50ms, P95 < 200ms, P99 < 500ms
- Curator read operations (no expansion): P50 < 20ms, P95 < 100ms
- Curator read operations (with code expansion): P50 < 50ms, P95 < 200ms
- Code registry lookup: P50 < 10ms, P95 < 50ms

### 8.2 Expanded Database Coverage

The compressed coding system applies to all five databases in the Curator roster, not only FalkorDB and the vector store.

**curator-doc (PostgreSQL Document Store)**

Document metadata fields are candidates for compressed codes:
- `doc_type` → domain-category code (e.g., `TEC-INFR` instead of full-text document type descriptions)
- `source_category` → code from RES domain (e.g., `RES-ACAD` for academic sources)
- `processing_status` → code from OPS-VLDT (e.g., `OPS-VLDT-PASS`)

Benefits: smaller row sizes, faster index scans, predictable WHERE clause matching, same disambiguation guarantee as FalkorDB.

**curator-state (Temporal Workflow State)**

Temporal's custom search attributes carry pipeline metadata that benefits from compressed codes:
- Gate identifiers: `G3` instead of "Entity deduplication gate"
- Error categories: `ERR-DEDUP` instead of full error descriptions
- Pipeline identifiers: codes from OPS-PIPE domain

Benefits: faster workflow search and filtering; consistent vocabulary across workflow queries.

**curator-config (YAML/Config Store)**

Configuration values referencing domains, categories, or components use compressed codes:
- Routing targets: `curator-kg` (already short, but domain references within configs use codes)
- Domain filters in pipeline configs: `TEC` instead of "Technology"
- Feature flags tied to specific capabilities: codes from relevant domains

Benefits: consistency across all configuration files; same vocabulary everywhere.

## APPENDIX B: W3C PROV-O MAPPING

The W3C PROV ontology (PROV-O) provides a formal framework for describing provenance. It defines three core classes and a set of relationships between them. The Amplified attribution record maps directly to these concepts.

---

### B.1 Core PROV-O Classes

| PROV-O Class | Definition | Amplified Equivalent |
|--------------|------------|---------------------|
| `prov:Entity` | A physical, digital, conceptual, or other kind of thing with some fixed aspects | Any data object: a finding, a vault document, a graph node, a report, an embedding |
| `prov:Activity` | Something that occurs over a period of time and acts upon or with entities | Any pipeline operation: a search, a transform, a validation, an ingest, an embed |
| `prov:Agent` | Something that bears some form of responsibility for an activity taking place | Any actor: a named pipeline step, an LLM model, a human operator, a Curator Agent |

---

### B.2 Core PROV-O Relationships

| PROV-O Relationship | Direction | Amplified Usage |
|--------------------|-----------|-----------------|
| `prov:wasGeneratedBy` | Entity → Activity | A finding was generated by a search activity |
| `prov:used` | Activity → Entity | A transform activity used an input document |
| `prov:wasAssociatedWith` | Activity → Agent | A search activity was associated with the research pipeline agent |
| `prov:wasAttributedTo` | Entity → Agent | A node in the knowledge graph was attributed to curator-kg |
| `prov:wasDerivedFrom` | Entity → Entity | An enriched document was derived from the original ingested document |
| `prov:actedOnBehalfOf` | Agent → Agent | The research pipeline acted on behalf of the human operator |
| `prov:wasInformedBy` | Activity → Activity | The entity resolution activity was informed by the label extraction activity |

---

### B.3 Attribution Record to PROV-O Mapping

The Amplified attribution record YAML structure maps to PROV-O as follows:

| Attribution Field | PROV-O Concept | Notes |
|-------------------|---------------|-------|
| `attribution.created_by.agent` | `prov:Agent` (software agent) | The pipeline step or service identifier |
| `attribution.created_by.human` | `prov:Agent` (person) | Human agent, relates to software agent via `prov:actedOnBehalfOf` |
| `attribution.created_by.model` | `prov:Agent` (software agent) | The LLM as a contributing agent |
| `attribution.derived_from[].source` | `prov:Entity` (source entity) | The entity from which the current entity was derived |
| `attribution.derived_from[]` (relationship) | `prov:wasDerivedFrom` | The derivation relationship between entities |
| `attribution.transforms[].operation` | `prov:Activity` | The named transform operation |
| `attribution.transforms[].agent` | `prov:Agent` | The agent responsible for the transform activity |
| `attribution.transforms[].timestamp` | `prov:startedAtTime` / `prov:endedAtTime` | Temporal bounds of the activity |
| `attribution.transforms[].gate` | `prov:qualifiedAssociation` | Gate result as a qualified property of the association |

---

### B.4 PROV-O Bundle Concept

PROV-O introduces the concept of a **bundle** — a named set of provenance assertions that can itself be attributed. In Amplified terms, a Curator's audit log for a single operation session is a bundle: a named, attributed collection of provenance statements.

This means the Curator's audit log is not just an operational log — it is a formal PROV-O bundle, and is itself attributable. The audit log knows who created it (the Curator), when, and what operations it records. This property is what makes the audit trail itself auditable: you can verify that the audit log has not been tampered with by checking the bundle attribution.

---

### B.5 Practical Implication

The formal PROV-O mapping means that Amplified attribution records can be:

1. **Exported to any PROV-O compatible system** without translation — the concepts map directly
2. **Queried using SPARQL PROV extensions** if a future architecture introduces an RDF layer
3. **Validated against the PROV formal constraints** — PROV-O defines rules about temporal consistency (an entity cannot be used before it was generated) that provide an additional layer of attribution integrity checking

The Amplified attribution system is not a bespoke format. It is a domain-specific application of an internationally standardised provenance framework. This is by design. When we say we stand on the shoulders of giants, this is one of those giants.
