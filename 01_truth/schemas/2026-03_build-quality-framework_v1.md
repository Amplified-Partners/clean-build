**1. What This Document Is**

This is the master framework for how Amplified Partners builds its operating system. Every process in the entire corpus of work gets subjected to a six-stage pipeline:

1.  Decomposed into atomic units

2.  Scored 0--10 on a defined rubric

3.  Researched against best-in-class (the \"gold standard\")

4.  Tested (Kaizen testing, Chaos testing, logic checking)

5.  Reassembled with symbiotic relationships identified

6.  Radically attributed (every claim, quote, and data point traced to source)

This framework draws from several established methodologies and combines them with Amplified\'s own Pudding Technique --- the Swanson ABC model (1986) adapted for business knowledge synthesis by Ewan Bramley and Claude (2026). The result is a unified quality system that doesn\'t exist elsewhere as a single framework.

The pipeline subtitle --- Decompose → Score → Research → Test → Reassemble --- is the operating rhythm. Every atom of work passes through it. No exceptions.

**3. The Decompose-Score-Research-Test-Reassemble Pipeline**

The following six-stage pipeline is the core operating rhythm. Every process, every claim, every atom of work passes through it sequentially. Each stage has defined inputs, outputs, and quality criteria.

**Stage 1: Decompose**

-   Break the entire corpus into atomic units: processes, claims, quotes, data points, SOPs, principles

-   Each atom gets a unique ID

-   Each atom gets a PUDDING 2026 label: WHAT.HOW.SCALE.TIME

-   Each atom gets tagged with APQC PCF category (1--13)

-   Each atom gets Bounded or Creative classification

**Stage 2: Score (0--10)**

-   Every atom scored on a 0--10 scale against defined criteria (see Section 4)

-   The scoring rubric is universal --- same rubric for every process in the system

-   Score represents current maturity/readiness

-   Baseline score established before any improvement

**Stage 3: Research**

-   For each process/atom, research the \"gold standard\" --- best-in-class from anywhere in the world

-   Use triple search: success patterns AND failure patterns

-   Collect named techniques, not just general findings

-   Apply Pudding Technique: strip domain labels, find cross-domain bridges

-   Document what \"10/10\" looks like for each process

**Stage 4: Test**

-   **Kaizen Testing:** incremental improvement cycles (PDCA)

-   **Chaos Testing:** deliberately break things to find weaknesses

-   **Logic Testing:** check for logical consistency, contradictions, gaps

-   Always updating, always checking --- but don\'t implement changes recklessly. Check logic and patterns FIRST.

-   Quality gates between departments: nothing advances without passing defined criteria

**Stage 5: Reassemble**

-   Bring tested atoms back together

-   At each scale (atom → module → department → system), look for patterns

-   Identify symbiotic relationships (Pudding cross-domain bridges)

-   Map information flow: what needs to tell what, what information affects what

-   Identify economisation opportunities: where the same mechanism serves multiple purposes

-   Test the reassembled whole (integration testing)

**Stage 6: Attribute**

-   Every atom carries radical attribution: source, author, confidence, fact percentage

-   Uses TASL format (Title, Author, Source, License)

-   Checked through the lens of: Radical Honesty, Radical Transparency, Win-Win, Whitehead principles

-   Layer 0 Laws compliance check on every output

**4. The 0--10 Process Scoring Rubric**

This is the universal scoring rubric for EVERY process in the system. No exceptions. The same rubric applies whether you\'re scoring a financial reconciliation SOP or a content creation workflow.

  ----------- ------------------- ---------------------------------------------------------------------------------------------------------------------
  **Score**   **Label**           **Criteria**
  **0**       **Non-existent**    Process not defined. No documentation. No one knows how it works.
  **1**       **Ad-hoc**          Process exists in someone\'s head. Tribal knowledge only.
  **2**       **Informal**        Process documented loosely (notes, messages). Not standardised.
  **3**       **Defined**         Process written as SOP. Steps documented. But not tested.
  **4**       **Measured**        Process has baseline metrics. We know how it currently performs.
  **5**       **Researched**      Best-in-class benchmark identified. We know what \"perfect\" looks like. Gap analysis done.
  **6**       **Improved**        Process redesigned based on research. Changes implemented.
  **7**       **Tested**          Kaizen tested. Chaos tested. Logic verified. Edge cases checked.
  **8**       **Verified**        Passed quality gates. Integration tested with adjacent processes. No contradictions.
  **9**       **Attributed**      Fully attributed. Every claim sourced. Radical transparency applied. Ready for audit.
  **10**      **Gold Standard**   Best-in-class. Continuously monitored. Kaizen loop active. Proven in production. Pudding\'d for cross-domain value.
  ----------- ------------------- ---------------------------------------------------------------------------------------------------------------------

**Thresholds**

-   **Pre-ship minimum:** Every process must score ≥ 7 before product ships.

-   **Target:** Every process at 9+ within 90 days of ship.

-   **Gold standard:** Score 10 is never permanent --- it triggers the next Kaizen cycle. If you think you\'re done, you\'re not looking hard enough.

**5. Data Size Estimation and Mitigation**

This section addresses the data growth concern directly. The short answer: Beast is massively over-provisioned for SMB scale. The long answer follows.

**Per-Client Data Model (FalkorDB Tenant)**

  **Metric**                    **Nodes**                 **Relationships**
  Initial onboarding            \~5,000--15,000           \~20,000--60,000
  Monthly growth                \~500--2,000              \~2,000--8,000
  Year 1 per client             \~25,000--40,000          \~100,000--160,000
  Storage per client (Year 1)   \~50--200MB in FalkorDB   Graphs are storage-efficient
  ----------------------------- ------------------------- ------------------------------

**Scale Projections**

  ----------------- ---------------------- --------------------------------
  **Scale**         **Total Graph Data**   **Notes**
  **10 clients**    \~2GB total            Trivial for Beast
  **50 clients**    \~10GB total           Well within capacity
  **200 clients**   \~40GB total           Still within Beast\'s capacity
  ----------------- ---------------------- --------------------------------

Beast has 256GB RAM (96GB allocated to Ollama, \~8GB to FalkorDB currently). FalkorDB can handle millions of nodes per graph with sub-10ms queries. At 200 clients, still well within Beast\'s capacity.

**The Real Growth Concern Is NOT Graph Data**

The actual storage pressure comes from four sources:

1.  Vector embeddings (Qdrant) --- approximately 4× the size of source text

2.  LLM trace logs (Langfuse) --- grows with every API call

3.  Content generation artifacts (MinIO) --- videos, images, audio

4.  Vault documents --- the raw corpus itself

**Mitigation Strategies**

1.  Tiered storage: hot (FalkorDB/Redis) → warm (PostgreSQL) → cold (MinIO/S3)

2.  Embedding pruning: re-embed on schedule, drop stale vectors

3.  Log rotation: Langfuse traces archived after 90 days

4.  Content lifecycle: generated content archived after campaign ends

5.  Per-client storage budgets: alert at 80% of tier allocation

6.  Federated graph (Tier 3): anonymised aggregates only, never full client data

**Verdict**

**The data will NOT be \"too fucking big\" for Beast at SMB scale.** The 10Gbit pipe and 256GB RAM are massively over-provisioned for the first 50--100 clients. The bottleneck will be compute (LLM API calls), not storage. Ewan\'s instinct that \"it\'s going to be too small\" is closer to correct for at least the first 18 months.

If Amplified scales to 500+ clients or moves to enterprise, THEN consider:

-   FalkorDB clustering (multiple primaries + replicas)

-   Dedicated Hetzner boxes per region

-   Separate Ollama inference nodes

**7. The Mathematical Formula for Objective Scoring**

Ewan asked for an objective mathematical formula. Here it is.

**Process Readiness Score (PRS)**

**PRS = (D × 0.15) + (M × 0.15) + (R × 0.20) + (T × 0.25) + (A × 0.10) + (I × 0.15)**

Where each variable is scored 0--10:

  --------- ---------------------------- ------------ -------------------------------------------
  **Var**   **Dimension**                **Weight**   **Rationale**
  **D**     Documentation completeness   **0.15**     Is the process fully defined?
  **M**     Measurement maturity         **0.15**     Do we have baseline metrics?
  **R**     Research depth               **0.20**     Have we found and analysed best-in-class?
  **T**     Testing rigour               **0.25**     Kaizen + Chaos + Logic testing complete?
  **A**     Attribution completeness     **0.10**     All claims sourced and verified?
  **I**     Integration fitness          **0.15**     Does it work with adjacent processes?
  --------- ---------------------------- ------------ -------------------------------------------

**Weighting Rationale**

Testing gets highest weight (0.25) because untested processes are the highest risk. Research gets 0.20 because knowing what \"good\" looks like is prerequisite to improvement. Integration gets 0.15 because isolated perfection is worthless. Attribution gets 0.10 --- important but not a blocker for functionality.

**Thresholds**

-   **Ship threshold:** PRS ≥ 7.0

-   **Gold standard threshold:** PRS ≥ 9.0

**Compound Readiness Score (CRS)**

For a module or department, aggregate individual PRS scores:

**CRS = (Σ PRS\_i × w\_i) / Σ w\_i**

Where w\_i is the criticality weight of each process. Bounded processes are weighted 1.5× (they\'re the load-bearing walls). Creative processes are weighted 1.0× (important but more flexible).

**8. Pudding Integration Points**

Where the Pudding Technique slots into this framework:

**1. Stage 3 (Research)**

When researching gold standard for a process, apply PUDDING labelling to all techniques found. Strip domain names. Look for cross-domain bridges. A finance technique might have an identical structural signature to a marketing technique --- the labels reveal this.

**2. Stage 5 (Reassemble)**

When reassembling atoms into modules, run a Pudding session on the module. Look for symbiotic relationships between processes that share PUDDING labels but come from different departments.

**3. Pattern Detection at Scale**

At each scale level (atom → module → department → system), look for identical or near-identical PUDDING labels. These are economisation opportunities --- where one mechanism can serve multiple purposes.

**4. The \"Little Bits into Big Bits\" Pattern**

This is exactly Swanson ABC. Small observations from different domains, when bridged, reveal system-level insights. The mathematical validation (Part 13 of the Pudding skill) proves this is statistically significant (p \< 0.001 for 4/4 label matches across domains).

**5. Death Spiral Detection (Reverse Pudding)**

Same engine, different rubrics. Instead of looking for positive emergence (5+5=13), look for negative emergence --- two falling metrics that together signal something worse than either alone. This is how the system catches cascading failures before they propagate.

**9. Department Boundaries as Quality Gates**

Ewan\'s description of \"departments that don\'t allow stuff in without being tested\" formalises as quality gates. Each boundary has defined criteria and pass conditions:

  ----------------------------- ------------------------------------------------- -------------------------------------------------------------
  **Boundary**                  **Gate Criteria**                                 **Pass Condition**
  **Raw → Decomposed**          Atom has unique ID, PUDDING label, APQC tag       All three present
  **Decomposed → Scored**       PRS baseline established                          Score recorded, not just estimated
  **Scored → Researched**       Gold standard identified, gap analysis complete   Gap document exists with sources
  **Researched → Tested**       Kaizen + Chaos + Logic tests executed             All tests pass or failures documented with remediation plan
  **Tested → Attributed**       Full TASL attribution, fact %, confidence band    Layer 0 Laws compliance check passed
  **Attributed → Production**   PRS ≥ 7.0, integration test passed                No unresolved contradictions with adjacent processes
  ----------------------------- ------------------------------------------------- -------------------------------------------------------------

**Enforcement**

The Enforcer agent (GPT-4.1-mini, 10-minute cron) checks gate compliance. Nothing advances without the gate check passing. Three consecutive failures trigger escalation to Ewan.
