## 1. Project Overview

### 1.1 Purpose
Today Mirror is a Mac-first daily task assistant that helps users align their actual behavior with their stated intentions through:
- **Constraint:** Maximum 3 visible tasks per day (no more)
- **Reflection:** Left/right layout showing "intended" vs "done"
- **Modes:** Three behavioral frameworks (Money-First, Balance, Recovery)
- **Privacy:** 100% local storage and local LLM processing
- **Honesty:** Factual feedback only, no motivational manipulation

### 1.2 Core Philosophy
**Behavioral Science, Not Gamification**
- Show what actually happened, not what should have happened
- Use micro-wins (visual cues) instead of praise text
- Daily celebration summaries are factual, not judgmental
- Weekly pattern mirrors for reflection, not daily guilt
- The app's job: show reality and help align Today with stated goals

### 1.3 Anti-Patterns (Forbidden)
- ❌ No pep-talks or motivational messages
- ❌ No gamification (points, badges, streaks)
- ❌ No dark patterns (guilt, shame, manipulation)
- ❌ No cloud AI services (Claude, GPT, etc.)
- ❌ No popups or modal dialogs for task completion
- ❌ No more than 3 tasks visible in "Today"

## 2. Data Model

### 2.1 Task
```typescript
interface Task {
  id: string;                    // UUID
  title: string;                 // User-facing description
  lane: 'revenue' | 'delivery' | 'life';  // Which lane this belongs to
  status: 'intended' | 'done' | 'archived';
  created_at: string;            // ISO 8601 timestamp
  completed_at?: string;         // ISO 8601 timestamp (if done)
  archived_at?: string;          // ISO 8601 timestamp (if archived)
  source: 'manual' | 'llm_suggested';  // How it was created
}
```

**Rules:**
- Maximum 3 tasks with `status: 'intended'` at any time
- When a 4th task is proposed, user must choose which existing task to replace or archive
- Completed tasks move to `status: 'done'` and appear in the Done strip
- Tasks can be archived (removed from Today without completion)

### 2.2 Interaction
```typescript
interface Interaction {
  id: string;                    // UUID
  timestamp: string;             // ISO 8601 timestamp
  user_input: string;            // Raw text from "What's on your mind?"
  llm_response: {
    cleaned_text: string;        // Cleaned/normalized version
    tasks: Task[];               // Extracted task suggestions
    confidence: number;          // 0-1 confidence score
  };
  action_taken: 'added' | 'ignored' | 'modified';
  notes?: string;                // Optional user notes
}
```

**Rules:**
- Every LLM interaction is logged
- User can review past interactions
- Interactions are stored in both JSON and Obsidian markdown

### 2.3 DailyLog
```typescript
interface DailyLog {
  date: string;                  // YYYY-MM-DD
  mode_set: 'money_first' | 'balance' | 'recovery';
  mode_actual: 'money_first' | 'balance' | 'recovery' | 'mixed';
  tasks_committed: Task[];       // Tasks that were in "intended" at start of day
  tasks_completed: Task[];       // Tasks that reached "done" status
  tasks_archived: Task[];        // Tasks removed without completion
  interactions: Interaction[];   // All LLM interactions for the day
  summary_text: string;          // Factual summary (no praise/guilt)
  pattern_notes?: string;        // Optional reflection notes
}
```

**Rules:**
- One DailyLog per calendar day
- Generated at end of day or on demand
- `mode_actual` is calculated from completed tasks' lanes
- Summary uses factual language only (see Section 5.3)

## 3. Behavioral Modes

### 3.1 Mode Definitions

#### Money-First Mode
**Purpose:** Revenue generation and business growth  
**Lane Rules:**
- `revenue`: 2 slots (66%)
- `delivery`: 1 slot (33%)
- `life`: 0 slots (0%)

**When to Use:** Building business, financial pressure, growth phase

#### Balance Mode
**Purpose:** Sustainable productivity across all areas  
**Lane Rules:**
- `revenue`: 1 slot (33%)
- `delivery`: 1 slot (33%)
- `life`: 1 slot (33%)

**When to Use:** Steady state, maintaining equilibrium

#### Recovery Mode
**Purpose:** Rest, health, personal needs  
**Lane Rules:**
- `revenue`: 0 slots (0%)
- `delivery`: 1 slot (33%)
- `life`: 2 slots (66%)

**When to Use:** Burnout prevention, health issues, personal crisis

### 3.2 Mode Enforcement
- User selects mode at start of day (or uses previous day's mode)
- When adding a task, app validates against mode's lane rules
- If lane is full, user must replace an existing task in that lane
- Mode can be changed mid-day, but triggers a warning about commitment

### 3.3 Mode Calculation (Actual)
At end of day, calculate `mode_actual` based on completed tasks:
```typescript
function calculateActualMode(completed: Task[]): string {
  const lanes = completed.map(t => t.lane);
  const revenue = lanes.filter(l => l === 'revenue').length;
  const delivery = lanes.filter(l => l === 'delivery').length;
  const life = lanes.filter(l => l === 'life').length;
  
  // Match to closest mode pattern
  if (revenue >= 2) return 'money_first';
  if (life >= 2) return 'recovery';
  if (revenue === 1 && delivery === 1 && life === 1) return 'balance';
  return 'mixed';  // Doesn't match any mode
}
```

### 4.2 Visual Behavior

#### Task Completion (Micro-Win)
When user checks a task:
1. Checkbox animates to checkmark (✓)
2. Task text gets subtle strikethrough
3. Task card background changes to muted color
4. After 0.5s delay, task smoothly slides to right column (Done strip)
5. Done strip shows task in same row position
6. **No popup, no toast, no text praise**

#### Task Addition
When adding a 4th task:
1. Show inline warning: "Today is full. Replace a task or archive one first."
2. Highlight the 3 existing tasks
3. User clicks one to replace or archive
4. New task takes that position

#### Mode Selector
- Dropdown in top-right
- Shows current mode with icon
- Changing mode shows warning if tasks don't match new rules
- Offers to archive non-compliant tasks

## 5. Local LLM Integration

### 5.1 Endpoint
**URL:** `http://localhost:11434/api/generate`  
**Model:** User's choice (Ollama default)  
**Fallback:** If endpoint unavailable, show dummy response for testing

### 5.2 Request Format
```json
{
  "model": "llama3.2",
  "prompt": "Extract actionable tasks from this text. Return JSON with cleaned_text and tasks array. Each task needs title and lane (revenue/delivery/life).\n\nUser input: {user_input}",
  "stream": false,
  "format": "json"
}
```

### 5.3 Expected Response
```json
{
  "cleaned_text": "Normalized version of user input",
  "tasks": [
    {
      "title": "Task description",
      "lane": "revenue",
      "confidence": 0.85
    }
  ]
}
```

### 5.4 Error Handling
- If LLM unavailable: Show "LLM offline. Add tasks manually."
- If response malformed: Log error, show "Couldn't parse response. Try again."
- If confidence < 0.5: Mark task as "low confidence" in UI
- Always allow manual task creation as fallback

## 7. Daily Summary & Weekly Patterns

### 7.1 Daily Summary Generation
**Trigger:** User clicks "Generate Today's Summary" or automatic at 11:59 PM

**Process:**
1. Aggregate all tasks from today
2. Calculate `mode_actual` from completed tasks
3. Generate factual summary text
4. Write to `daily_logs.json` and Obsidian markdown
5. Show summary in UI (non-modal, inline)

**Summary Template (Factual Only):**
```
You committed to {N} tasks in {mode_set} mode.
You completed {M} tasks ({breakdown by lane}).
You archived {K} tasks.
Your actual mode was {mode_actual}.
```

**Examples:**
- "You committed to 3 tasks in Money-First mode. You completed 2 tasks (1 revenue, 1 delivery). You archived 1 task. Your actual mode was Mixed."
- "You committed to 3 tasks in Balance mode. You completed 3 tasks (1 revenue, 1 delivery, 1 life). Your actual mode matched your intended mode."

### 7.2 Weekly Pattern Mirror (v0.2 Feature)
**Trigger:** Sunday evening or on-demand

**Process:**
1. Aggregate last 7 daily logs
2. Calculate patterns:
   - Most common mode set
   - Most common mode actual
   - Mode alignment rate (set vs actual)
   - Completion rate by lane
   - Total tasks committed vs completed
3. Generate factual pattern summary
4. Write to Obsidian `weekly_patterns/` folder

**Pattern Template:**
```
Week of {start_date} to {end_date}

Mode Patterns:
- Most set: {mode} ({N} days)
- Most actual: {mode} ({N} days)
- Alignment: {X}% of days matched

Completion Patterns:
- Revenue: {X}% completion rate
- Delivery: {Y}% completion rate
- Life: {Z}% completion rate

Total: {N} tasks committed, {M} completed ({X}% rate)
```

## 8. Technical Stack (v0.1)

### 8.1 Option A: SwiftUI macOS App (Preferred)
**Pros:**
- Native Mac experience
- Better system integration
- Faster performance
- Smaller footprint

**Cons:**
- Steeper learning curve
- Mac-only (no web version)

**Stack:**
- SwiftUI for UI
- Swift for business logic
- FileManager for JSON storage
- URLSession for LLM HTTP calls

### 8.2 Option B: Local Web App (Fallback)
**Pros:**
- Easier to build quickly
- Cross-platform potential
- Familiar web technologies

**Cons:**
- Less native feel
- Requires local server

**Stack:**
- Frontend: React + TypeScript + Tailwind
- Backend: Node.js + Express (minimal)
- Storage: fs module for JSON
- LLM: fetch/axios for HTTP calls

### 8.3 Decision Criteria
- If confident in SwiftUI: Choose Option A
- If time-constrained or unfamiliar with Swift: Choose Option B
- **For v0.1 MVP:** Choose the stack that gets to working code fastest

## 16. References

### 16.1 Behavioral Science Principles
- **Choice Overload:** Limiting to 3 tasks reduces decision paralysis
- **Implementation Intentions:** Specific tasks increase completion rate
- **Commitment Devices:** Declaring mode creates accountability
- **Micro-wins:** Small victories build momentum without manipulation
- **Reflection:** Weekly patterns enable self-awareness without guilt

### 16.2 Design Principles
- **Hick's Law:** Fewer choices = faster decisions
- **Fitts's Law:** Large targets = easier interaction
- **Jakob's Law:** Familiar patterns = lower cognitive load

### 16.3 Anti-Patterns to Avoid
- **Dark Patterns:** Manipulation, guilt, shame
- **Gamification:** Points, badges, streaks (creates extrinsic motivation)
- **Notification Spam:** Constant interruptions reduce focus
